# 当前车牌识别模型总结

生成日期：2026-07-05

## 1. 项目目标

当前项目目标是实现中国车牌号自动识别，覆盖两类输入：

1. 裁剪后的车牌图片 -> 完整车牌号。
2. 整车图片 -> 检测车牌位置 -> 裁剪车牌 -> 识别完整车牌号。

项目已经从早期的单纯 CRNN 识别，扩展为“车牌检测 + 字符识别”的完整工程流程。目前工作区内保留了三类核心模型：

| 模型 | 权重文件 | 作用 | 当前定位 |
|---|---|---|---|
| YOLO 车牌检测器 | `models/plate_detector.pt` | 在整车图中定位车牌框 | 主流程检测模块 |
| CRNN + CTC 识别器 | `models/license_plate_crnn_best.pth` | 从裁剪车牌图直接输出整牌字符串 | 当前最可靠识别模块 |
| YOLO 字符检测器 | `models/char_detector.pt` | 检测裁剪车牌中的单个字符并排序拼接 | 对照实验与可视化辅助模块 |

## 2. 当前推荐流程

当前推荐以 YOLO 车牌检测器和 CRNN 识别器作为主流程：

```text
整车图片
-> YOLO 车牌检测
-> 裁剪车牌区域
-> CRNN + CTC 整牌识别
-> 输出车牌号码和可视化结果
```

原因是：车牌检测器的定位能力已经很强，而 CRNN 在整牌识别上明显优于 YOLO 字符检测方案。

纯 YOLO 方案也已经具备可运行能力：

```text
整车图片
-> YOLO 车牌检测
-> 裁剪车牌区域
-> YOLO 字符检测
-> 按 x_center 排序
-> 结合车牌规则后处理
-> 拼接车牌号码
```

但目前纯 YOLO 字符方案的整牌准确率仍低于 CRNN，适合作为错误分析、字符框可视化和后续优化方向。

## 3. 数据集情况

项目主要使用 CCPD 数据集。`data/ccpd/labels.csv` 当前样本划分如下：

| split | 样本数 |
|---|---:|
| train | 246,370 |
| val | 52,827 |
| test | 52,780 |
| 合计 | 351,977 |

车牌检测数据来自 CCPD 文件名中的车牌 bbox，转换为 YOLO 检测格式。当前主要检测训练集为：

```text
data/yolo_plate_20k/plate.yaml
```

字符检测数据来自 CCPD 车牌裁剪图和车牌字符串。字符类别共 65 类，包括：

- 31 个省份简称；
- 24 个字母，去除 I、O；
- 10 个数字。

需要注意：当前字符检测框不是人工标注框，而是按车牌宽度均分得到的伪字符框。这是 YOLO 字符识别效果受限的主要原因之一。

## 4. 模型结构与训练配置

### 4.1 YOLO 车牌检测器

车牌检测器使用 YOLOv8n 作为基础模型，训练目标只有一个类别：`license_plate`。

主要训练配置：

| 配置项 | 值 |
|---|---:|
| 训练目录 | `runs/detect/plate_detector_20k-3` |
| 数据配置 | `data/yolo_plate_20k/plate.yaml` |
| 基础模型 | YOLOv8n |
| 输入尺寸 | 416 |
| epoch | 50 |
| batch size | 8 |
| device | CPU |

最终部署权重：

```text
models/plate_detector.pt
```

### 4.2 CRNN + CTC 识别器

CRNN 模型用于裁剪车牌图到完整车牌字符串的识别。

模型结构包括：

- CNN 特征提取层；
- 双向 LSTM 序列建模层；
- 线性分类层；
- CTC 解码输出车牌字符串。

输入侧默认面向裁剪后的单车牌图，因此如果输入是整车图，需要先经过车牌检测和裁剪。

第八周完成了一轮新的 CRNN 微调，训练会话为 `plate_train_v2`。本轮微调将 batch size 从 32 提升到 64，将训练轮数从 20 扩展到 80，并采用 `StepLR(step=15, gamma=0.5)` 学习率衰减策略，即每 15 个 epoch 学习率减半。训练观察时 GPU 显存占用约 1991MB / 24564MB，利用率约 21%，当时仍处于数据加载阶段。

最新 CRNN 微调配置：

| 配置项 | 值 |
|---|---:|
| 训练会话 | `plate_train_v2` |
| batch size | 64 |
| epoch | 80 |
| 学习率衰减 | `StepLR(step=15, gamma=0.5)` |
| GPU 显存占用 | 1991MB / 24564MB |
| GPU 利用率 | 21% |

最终部署权重：

```text
models/license_plate_crnn_best.pth
```

### 4.3 YOLO 字符检测器

字符检测器同样使用 YOLOv8n，目标是在裁剪后的车牌图上检测每个字符框并预测字符类别。

当前最新训练目录：

```text
runs/detect/yolo_chars_v2
```

主要训练配置：

| 配置项 | 值 |
|---|---:|
| 数据配置 | `data/yolo_chars/chars.yaml` |
| 基础模型 | YOLOv8n |
| 输入尺寸 | 160 |
| epoch | 20 |
| batch size | 8 |
| 类别数 | 65 |
| device | CPU |

最终部署权重：

```text
models/char_detector.pt
```

后处理逻辑位于 `src/recognize_plate_yolo.py`，主要包括：

- 按字符框 `x_center` 从左到右排序；
- 对中心点过近或 IoU 较高的重复框去重；
- 候选字符数限制为 7 或 8；
- 引入车牌位置规则：第 1 位为省份，第 2 位为字母，后续为字母或数字；
- 使用置信度、空间间距和位置合法性综合评分选择最终字符序列。

## 5. 当前性能表现

### 5.1 YOLO 车牌检测器

`plate_detector_20k-3` 在验证集上的主要指标如下：

| 指标 | 最佳 epoch | 最佳值 |
|---|---:|---:|
| precision(B) | 34 | 99.887% |
| recall(B) | 17 | 99.887% |
| mAP50(B) | 28 | 99.485% |
| mAP50-95(B) | 48 | 79.340% |

最后一个 epoch 的指标：

| 指标 | 值 |
|---|---:|
| precision(B) | 99.877% |
| recall(B) | 99.887% |
| mAP50(B) | 99.460% |
| mAP50-95(B) | 79.211% |

结论：车牌检测器已经可以稳定找到车牌。mAP50 非常高，说明“是否找到车牌”的能力较强；mAP50-95 低一些，说明在严格 IoU 下 bbox 精细程度仍有提升空间。

### 5.2 CRNN 识别器

第八周微调后，CRNN 取得了当前最好的识别结果：

| 指标 | 值 |
|---|---:|
| 字符准确率 | 99.7% |
| 整牌准确率 | 98%+ |

上一版 CRNN 评估记录如下，可作为本轮微调前的基线：

| 指标 | 值 |
|---|---:|
| 测试样本数 | 52,780 |
| 字符准确率 | 98.94% |
| 整牌准确率 | 94.43% |
| 平均编辑距离 | 0.0743 |
| 空预测数量 | 0 |

结论：CRNN 是当前最可靠的整牌识别模块。本轮 `plate_train_v2` 微调后，整牌准确率从约 94.43% 提升到 98% 以上，字符准确率提升到 99.7%，说明更长训练周期、更大 batch size 和阶段式学习率衰减明显增强了完整车牌字符串输出的稳定性。

| mAP50(B) | 76.668% |

已有端到端评估记录中，YOLO 字符检测方案在 CCPD test split 上结果为：

| 指标 | 值 |
|---|---:|
| 测试样本数 | 52,780 |

结论：YOLO 字符检测已经具备可用雏形，但整牌准确率仍明显低于 CRNN。主要问题不是单字符完全不可识别，而是车牌识别对每一位都敏感，一旦多检、漏检或任意一位分类错误，整牌就会判错。

## 6. 方案对比

| 方案 | 输入 | 输出 | 优点 | 主要问题 | 当前建议 |
|---|---|---|---|---|---|
| CRNN 单独识别 | 裁剪车牌图 | 整牌字符串 | 整牌准确率高，序列建模强 | 不能直接处理整车图 | 保留为主识别器 |
| YOLO 车牌检测 + CRNN | 整车图 | 整牌字符串 + bbox | 端到端流程完整，整体可靠 | 检测框偏移会影响识别 | 当前主流程 |
| YOLO 车牌检测 + YOLO 字符检测 | 整车图 | 字符框 + 整牌字符串 | 可解释性强，字符可视化好 | 多检、漏检、伪框噪声、整牌准确率低 | 对照实验和辅助分析 |

## 7. 当前主要问题

1. 端到端真实场景评估不足  
   CCPD 上检测和识别指标较好，但自摄图片的光照、角度、模糊、遮挡和压缩质量与 CCPD 有差异，需要建立独立真实测试集。

2. 车牌裁剪仍是矩形裁剪  
   当前检测后直接按矩形 bbox 裁剪，没有透视矫正。倾斜车牌、远距离车牌和边缘裁剪偏移会影响识别。

3. YOLO 字符检测标签质量有限  
   字符框来自宽度均分伪标注，不是真实字符边界，导致检测器容易学习到不精确的位置，出现多检和漏检。

4. 字符检测端到端错误集中在多字符  
   已有错误分析显示，多字符错误是 YOLO 字符方案最大问题，占整牌错误的大头。

5. 车牌类型覆盖仍有限  
   当前字符表覆盖常规省份、字母和数字，但对新能源车牌、特殊车牌、双层车牌等场景支持还不完整。

6. 多车牌场景暂未作为主流程支持  
   当前默认选择置信度最高的一个车牌框，尚未系统支持一张图中多车牌逐个识别。

## 8. 后续优化建议

优先级建议如下：

1. 建立端到端真实测试集  
   从自摄图片或实际应用场景中整理一批整车图，人工标注真实车牌号，统计“整车图 -> 最终车牌号”的准确率。

2. 在 YOLO 检测后加入透视矫正  
   如果能从 bbox 或关键点估计车牌四角，可将倾斜车牌矫正为规则矩形，再送入 CRNN 或字符检测器。

3. 保持 YOLO + CRNN 为主流程  
   在当前阶段，不建议用 YOLO 字符检测替换 CRNN。CRNN 的整牌准确率明显更高，更适合作为稳定输出模块。

4. 改进 YOLO 字符检测数据  
   如果继续推进纯 YOLO 字符方案，应优先获取真实字符级 bbox，或通过更可靠的字符切分方法生成高质量伪标签。

5. 增加多车牌支持  
   将 `detect_best_plate` 从“只取最高置信度框”扩展为返回多个候选框，并对每个车牌独立裁剪和识别。

6. 针对错误类型做专项优化  
   对 D/0、B/8、Q/0、Z/7、T/7 等形近字符增加难例样本；对省份位和第二位字母加入更强的位置约束。

## 9. 运行入口

单独检测车牌：

```bash
python src/detect_plate_yolo.py \
  --image data/self_photos/raw/car1.jpg \
  --detector models/plate_detector.pt \
  --output data/self_photos/cropped/car1_plate.jpg \
  --conf 0.25
```

YOLO 车牌检测 + YOLO 字符检测：

```bash
python src/recognize_plate_yolo.py \
  --image data/self_photos/raw/car1.jpg \
  --plate-detector models/plate_detector.pt \
  --char-detector models/char_detector.pt \
  --conf 0.25
```

YOLO 车牌检测 + CRNN 识别：

```bash
python src/detect_and_recognize.py \
  --image data/self_photos/raw/car1.jpg \
  --detector models/plate_detector.pt \
  --recognizer models/license_plate_crnn_best.pth \
  --output results/detections/car1_result.jpg \
  --conf 0.25 \
  --device auto
```

## 10. 总结

当前项目已经形成了完整的车牌识别系统。YOLO 车牌检测器负责从整车图中定位车牌，CRNN 识别器负责输出完整车牌号，两者组合是目前最推荐的主流程。

从已有指标看，车牌检测器在验证集上 mAP50 约 99.46%，CRNN 在第八周 `plate_train_v2` 微调后整牌准确率达到 98% 以上、字符准确率达到 99.7%。这说明系统在标准 CCPD 场景下已经具备较好的可用性，识别模块也完成了一次明显升级。

YOLO 字符检测器作为纯 YOLO 识别方向已经完成初步验证，最新 `yolo_chars_v2` 的 mAP50 达到 76.668%，但端到端整牌准确率仍低于 CRNN。后续若要提升纯 YOLO 字符方案，应优先解决字符框伪标签质量、多检漏检和序列规则建模问题。

整体来看，下一阶段最值得投入的是：真实场景端到端评估、车牌透视矫正、多车牌支持，以及基于错误样本的专项优化。
